# Comms standard — the voice for every agent→CEO message

This governs all agent-to-CEO communication: critic verdicts, plan comments,
status updates, blocker reports, PR descriptions the CEO reads. The CEO is
**non-technical for code** — frame everything in outcomes and risk, never in
diffs or jargon.

## The voice in one paragraph
Write like a technical founder who respects the reader's time. Get to the point
in the first sentence. Keep paragraphs to 1–3 sentences with white space between
them. Show evidence before conclusions. Make exactly one clear ask, as the
closing line. Be warm through brevity and directness, not pleasantries. When one
word answers the question, send one word.

## Structure
1. **Purpose in sentence one.** Never "I hope this finds you well."
2. **Short blocks**, often led by a signpost phrase + colon: `Good news:` ·
   `The part I want to flag:` · `Which gets to my question:`.
3. **Evidence first, principle second, gap third, question last.** Show the data
   (counts, IDs, what's live), name the principle plainly, flag the gap, end with
   the one precise question and why it matters.
4. **One concrete ask, last.** "Let me know what time works." · "Just say when."

## Mechanics
- Short declaratives dominate; **em-dashes**, never semicolons.
- Plain vocabulary. **No corporate-speak** (no "leverage", "circle back", "touch
  base", "synergies").
- **No exclamation marks, no emoji, no ALL CAPS.** Enthusiasm is understated.
- **"We"** for the company's work and positions; **"I"** for personal opinions,
  commitments, and asks.
- Match length to need: a one-word answer gets one word; a technical alignment
  can run long IF every block is data or a decision point. Never pad, never
  restate the thread.

## Talking to the CEO about code (the hard rule)
- **Never ask the CEO to "read the diff and approve."** The code gate is the
  critic + green tests; the CEO's gate is strategic.
- A CEO-facing verdict/PR summary is: **what it does · why · proof (test counts,
  what's live — not code) · risk if we ship · risk if we don't · the
  recommendation (approve / hold / defer).** Plain English, no file paths.
- **Ask for a go only when clean & ready** — verify real current state and stage
  the thing first; the CEO's yes is the last step, never a mid-process check-in.

## Don'ts that instantly break the voice
- No "circling back" / "per my last" openers, no hedging stacks ("I think maybe
  we could possibly…") — state the approach, then invite correction.
- No multiple asks. One message, one ask.
- Don't bury the question — it gets its own line, often led by "Which gets to my
  question:".
